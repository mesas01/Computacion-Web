DESPLIEGUE

En esta sección se describe el proceso seguido para desplegar la aplicación en un entorno que simula producción, garantizando su funcionamiento adecuado, accesibilidad y estabilidad.

Control de versiones

Durante el desarrollo del proyecto se utilizó Git como sistema de control de versiones, permitiendo la gestión eficiente del código fuente, colaboración entre los miembros del equipo y seguimiento de los cambios realizados.

Se manejaron distintas ramas para organizar el trabajo:

  • main: Contiene el código más estable y preparado para producción.
  • DEV: Rama destinada a la integración de nuevas funcionalidades antes de ser fusionadas con main.
  • Ramas de cada integrante: Se crearon para que cada uno de los integrantes desarrolle en alguna funcionalidad del sistema por separado.

Se realizaron merges controlados, revisiones de código (pull requests) y pruebas antes de aprobar los cambios. Esto garantizó que el despliegue final se hiciera con una versión probada, limpia y estable del sistema.

Se utilizó GitHub como plataforma para alojar el repositorio, permitiendo el trabajo colaborativo, la gestión de issues y la integración con pipelines de despliegue.

Entorno de despliegue

Se utilizó un entorno proporcionado por la Universidad, el cual permite simular condiciones reales de producción. Este entorno cuenta con soporte para tecnologías como Docker y Kubernetes, lo que facilita la orquestación y ejecución de contenedores de manera eficiente y escalable.

Herramientas utilizadas

  • Docker: Se usó para contenerizar tanto el frontend (Angular) como el backend (Spring Boot), asegurando que cada componente se ejecute de forma aislada y con sus dependencias correctamente definidas.
  • Kubernetes: Se utilizó para gestionar la orquestación de contenedores, facilitando el escalado, balanceo de carga y monitoreo de los servicios desplegados.
  • Docker Compose: Se empleó durante la etapa de desarrollo local para facilitar el levantamiento de servicios de forma conjunta antes del despliegue en Kubernetes.

Proceso de despliegue

  1. Se generó una imagen Docker para el backend y otra para el frontend.
  2. Las imágenes se subieron a un registro privado/local.
  3. Se configuraron los archivos YAML para el despliegue en Kubernetes.
  4. Se aplicaron los manifiestos de Kubernetes en el clúster de la universidad.
  5. Se expusieron los servicios mediante un Ingress Controller para permitir el acceso desde el navegador.
  6. Se realizaron pruebas en el entorno de despliegue para validar el correcto funcionamiento de la plataforma.

Validación del despliegue

Una vez desplegada la aplicación, se realizaron pruebas funcionales y de rendimiento en el entorno, asegurando que todas las funcionalidades estuvieran disponibles, que no hubiera errores en los logs de los contenedores y que el acceso a la plataforma fuera estable y rápido.
